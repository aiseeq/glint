Ключевые запреты и принципы
- GIT ТОЛЬКО ЧЕРЕЗ make commit: меняющие состояние git-команды — только с явного разрешения; read-only (`git status/diff/log`) — без ограничений. Принцип: "make commit by default, raw git only with permission".
- ЛОЖНЫЕ СРАБАТЫВАНИЯ ЧИНИТЬ В ПРАВИЛЕ: false positive → TDD-репро (красный тест с реальным кодом) → фикс правила; suppress в целевом проекте — только для документированных легитимных случаев, не вместо фикса правила. Принцип: "Fix the rule, don't silence the finding".
- Доверие пользователю: сообщил о проблеме — ищем, пока не найдена; не обесцениваем репорты.
- In-place правки: без «-backup/-v2/-old»; один источник истины; ненужное — удаляем, история в git.
- ЧИСТОЕ УДАЛЕНИЕ: без комментариев «removed/УДАЛЁН/was X»; deprecated aliases удалять сразу. Принцип: "Delete cleanly, git remembers".
- ЗАПРЕТ ТИХИХ ОШИБОК И НЕЯВНЫХ ФОЛБЕКОВ: после любой ошибки запрещён тихий/деградирующий фолбек (default/magic/0/""/nil/«угадать значение»). Ошибка всегда явная (сигнатура с error) и всегда залогирована; caller решает стратегию; неоднозначный ввод — тоже ошибка. Разрешены только явные фолбеки как часть дизайна — и они логируются. Принцип: "Fail explicitly and logged; fallbacks only when explicit by design".
- CANONICAL IMPLEMENTATION FIRST: перед написанием кода — grep-поиск существующей реализации (у правил много общей инфраструктуры: core.LineSuppresses, реестр, AST-хелперы); есть — использовать/рефакторить. Принцип: "Grep first, code second, duplicate never".
- ЕДИНАЯ ЛОГИКА НА ДЕЙСТВИЕ: на каждое действие — ровно одна функция/класс с общими тестами; второй участок с той же логикой — свести к единому источнику. Принцип: "One function per action, one source of truth, shared tests".
- ПОЧИНКА ВМЕСТО ОБХОДОВ: чиним причину, не ищем короткие пути. Принцип: "Fix root cause, not symptoms".
- ЗАПРЕТ ПРОПУСКА ПРОВЕРОК: падающие проверки (тесты, lint, golden) чинить, а не отключать; golden-файлы обновлять только осознанно, с проверкой diff. Принцип: "Fix the check failure, never skip the check".
- КРАТКИЕ РЕЗУЛЬТАТЫ РАБОТЫ: не создавать отдельные отчёты-документы — краткий итог итерации в текстовом выводе; все замеченные проблемы (даже побочные) обязательно сообщать в финале. Принцип: "Brief iteration results, all noticed problems reported".
- КРИТИЧНАЯ САМООЦЕНКА: критично анализировать свои результаты; нормально сказать «сделал не всё, ошибся, продолжу». Принцип: "Critical self-assessment over false confidence".
- КРИТИЧЕСКИЙ АНАЛИЗ ИЗМЕНЕНИЙ: после правки спросить себя: кто ещё вызывает? что если часть операции провалится? какие defaults/конфиги влияют? Сомнение — явно в вывод: «Потенциальная проблема: …». Принцип: "Doubt loudly, never silently assume correctness".
- NO-GUESSING DEBUGGING: сначала evidence (реальный вывод glint на тест-кейсе, AST-дамп, логи), потом гипотеза и фикс. Принцип: "Evidence first, hypothesis second, fix last".
- ПРИОРИТЕТ CLAUDE.MD НАРУШЕНИЙ: обнаружил нарушение принципов — остановиться, отложить задачу, устранить. Принцип: "CLAUDE.md violations are highest priority".
- ЗАПРЕТ LEGACY: legacy-код используется — рефакторить; лежит для совместимости — удалить. Принцип: "No legacy, only current code".

Анализ ошибок и тестирование
- HOOK ERRORS: при PreToolUse:Bash hook error — остановиться, прочитать ошибку, исправить поведение.
- RESEARCH-FIRST: перед исправлением проблемы или реализацией функциональности — официальная документация (go/ast, go/types, tree-sitter и т.п.) и WebSearch; готовые проверенные решения предпочтительнее самописных; trial-and-error без изучения docs запрещён. Принцип: "5 минут документации экономят часы отладки; официальный код > самописный".
- Root cause: выяснять первопричины, факт-чекинг реального состояния; спрашивать пользователя при неоднозначности.
- САМ СНАЧАЛА, ПОТОМ СПРОСИТЬ: вопрос о фактах сначала закрывать своими инструментами (grep, запуск glint на тест-кейсе, git log); пользователя спрашивать только о недоступном или о замысле. Принцип: "Exhaust all tools before asking a human".
- ВЕРИФИКАЦИЯ У ИСТОЧНИКА: утверждения из README/комментариев (количество правил, поведение) не копировать без проверки (`glint rules`, запуск); README вторичен относительно кода. Принцип: "Verify at source, never echo stale claims".
- САМОТЕСТИРОВАНИЕ: написал код/правило — сразу запустил тесты и self-check; отличия от ожидаемого явно в отчёт. Принцип: "Написал = сразу проверил; изменил = протестировал с честным отчётом".
- TDD ПРИ ИСПРАВЛЕНИИ БАГОВ: сначала тест, воспроизводящий баг (красный, с реальным кодом-репро из целевого проекта) → фикс → зелёный. Принцип: "Red test first, then fix, then green".

Версионирование и коммиты
- `make commit MESSAGE="..."` — stage all + commit; затем `git push`. Разрешён без запроса, так часто, как нужно; после задачи — коммит и push; при массовых правках — промежуточные коммиты.
- Raw git, меняющий состояние (`add/commit/reset/checkout/restore/stash/rebase/merge/cherry-pick`), — только с явного разрешения; read-only — без ограничений.
- Перед коммитом смотреть только на секреты и большие бинарники. Осмысленные MESSAGE с Conventional Commits.

Workflow агента
- Analysis → Design → Implementation → Validation: поиск дублей, типизированные структуры, явные ошибки, тесты; затем чек-лист самопроверки.
- Навигация: каждую bash-команду начинать с `cd /home/aisee/work/glint` (или полного пути к подпапке); никаких относительных путей без явного cd.
- НЕ LAUNCH-AND-IDLE: не запускать суб-агента или фоновую команду, если после запуска остаётся только ждать результата: сделать самому, распараллелить или активно мониторить. Принцип: "Never launch-and-idle — do it inline, parallelize, or actively monitor".
